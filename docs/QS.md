# 企业级 AI Agent 项目面试问题清单（仅问题）

标注说明：
- `【可答】`：可直接基于当前项目真实实现回答
- `【可答-需说明】`：可以回答，但需明确“当前为简化版/部分规划中”

## 第一类：架构设计类问题（18题）

1. 【可答-需说明】为什么选择 ReAct 模式而不是 Plan-and-Execute？
   - 项目现状版：为什么当前阶段采用“向量检索 + 固定模板输出”，而不是复杂 Agent（ReAct / Plan-and-Execute）工作流？
2. 【可答】采用多路检索还是单一向量检索？为什么
   - 真实回答：当前主链路是单一向量检索（pgvector Top-1），因为我们当前目标是先把“可用且稳定”的政务问答做深。考虑到本地模型和数据库都在一台机器上，单路向量链路延迟和可解释性都更可控。文本检索目前保留为可选兜底策略，但默认已切到严格向量模式。
3. 【可答-需说明】Agent 编排器是怎么设计的？为什么这样设计？
   - 项目现状版：当前系统没有复杂编排器，若后续引入多步骤工作流，编排层应如何演进？
4. 【可答-需说明】多 Agent 协作和单 Agent 有什么区别？你怎么选择？
   - 项目现状版：为什么当前先用单检索链路而非多 Agent 协作？升级条件是什么？
5. 【可答-需说明】意图识别模块是怎么实现的？
6. 为什么选择 Milvus 而不是其他向量数据库？
   - 项目现状版：为什么当前项目选择 PostgreSQL + pgvector，而不是 Milvus/Qdrant/ES 向量方案？
7. 【可答】系统的可扩展性是怎么设计的？
   - 真实回答：我把采集、导入、向量回填、调试检索都拆成独立脚本（`import_gxzwfw_services.py`、`backfill_embeddings.py`、`debug_vector_rank.py`），数据层按主表/材料/流程/向量表分层。配置全部走 `.env`，比如 provider、模型名、ivfflat probes 都可调，不需要改代码就能切运行策略。后续新增部门数据或替换 embedding 模型，基本都是配置和脚本层改动。
8. 【可答】如果让你重新设计这个系统，你会怎么改进？
   - 真实回答：我会做三件事：第一，加“相关性阈值+澄清问句”，避免低相关硬命中；第二，把检索升级为 ANN 召回 + 精排两阶段；第三，补全线上观测（QPS、P95、命中率、空结果率）并做自动化回归集。这样能同时提升准确率、稳定性和可运维性。
9. 【可答】你的系统能支持多少并发？怎么评估的？
   - 真实回答：当前是单机开发态，主要面向小规模并发。评估方式是压测 `/v1/chat`，分别测“冷启动首请求”和“热态请求”，同时观察数据库连接池、模型推理耗时和 API P95。我们还没做正式生产压测报告，所以会明确说明：目前目标是稳定正确性优先，不夸大并发指标。
10. 【可答-需说明】为什么要做模型路由？直接用一个模型不行吗？
    - 项目现状版：为什么当前先不做模型路由？未来在什么场景下需要引入模型路由？
11. 【可答-需说明】怎么设计的对话管理和上下文维护？
12. 【可答-需说明】你的系统支持哪些模型？怎么做模型适配的？
13. 【可答】如何处理长文档输入？上下文窗口不够怎么办？
   - 真实回答：当前服务端不走长对话生成，而是“问题 -> 向量检索 -> 固定模板返回”，所以天然规避了长上下文爆窗问题。对于长政务原文，我们在入库时做结构化拆分（事项主信息、材料、流程），查询阶段只取命中事项的关键字段输出，不把整篇原文塞进模型上下文。
14. 【可答-需说明】系统的安全架构是怎么设计的？
15. 【可答-需说明】你怎么看待 MCP（Model Context Protocol）？在你的系统中是怎么使用的？
16. 【可答】如何设计 Agent 的错误恢复和重试机制？
   - 真实回答：现在的恢复策略是分层降级：本地 embedding 不可用时给出明确错误或按配置回退；查询失败时返回可解释 fallback 文案；数据采集脚本支持 `--continue-on-error` 跳过坏样本继续跑。核心原则是“不静默失败”，每个阶段都可观测、可复现、可重跑。
17. 【可答-需说明】你的系统支持多租户吗？怎么实现的？
18. 【可答-需说明】你怎么看 Agent 架构的未来发展？

## 第二类：技术实现类问题（18题）

1. 【可答】向量数据库是怎么选型的？分块策略是什么？
   - 真实回答：我们选 PostgreSQL + pgvector，原因是部署简单、和现有关系数据天然一体化、开发调试成本低。索引用 `ivfflat`，当前 `lists=100`，并通过 `GOVFLOW_VECTOR_IVFFLAT_PROBES` 调召回。分块上不是通用文档切片，而是按政务语义结构化成“事项/材料/流程”三层，再把可检索文本写入 `service_embedding.vector_text`。
2. 【可答-需说明】ReAct 的 Prompt 是怎么设计的？
   - 项目现状版：当前“检索 + 模板输出”链路中，模板约束和可解释性是怎么设计的？
3. 【可答-需说明】熔断器是怎么实现的？三个状态是什么？
4. 【可答-需说明】记忆系统的短期和长期记忆是怎么配合的？
5. 【可答-需说明】文档解析支持哪些格式？怎么处理表格和图片？
6. 【可答-需说明】重排序是怎么做的？用的什么模型？
   - 项目现状版：当前为何先不引入独立 reranker？仅靠向量距离+阈值/规则的取舍依据是什么？
7. 【可答-需说明】Function Calling 和 MCP 有什么区别？你怎么选的？
8. 【可答-需说明】会话摘要压缩的阈值是多少？怎么确定的？
9. 【可答-需说明】RRF 融合排序的公式是什么？权重怎么调？
10. 【可答】嵌入模型用的什么？维度多少？怎么选的？
   - 真实回答：当前本地模型是 `BAAI/bge-base-zh-v1.5`，输出 768 维，和库里 `vector(768)` 一致。选它是因为中文语义表现稳定、生态成熟、可本地运行，且我们已完成全库向量回填。查询和回填保持同维度同模型，避免向量空间不一致。
11. 【可答-需说明】Guardrails（护栏）是怎么实现的？
12. 【可答-需说明】怎么实现的 Streaming 流式输出？
13. 【可答】怎么做的日志和链路追踪？
   - 真实回答：API 响应里会返回 `stages_executed`，明确走了哪些阶段（比如 `retrieve_vector`、`load_detail`、`template`）。离线侧有导入/回填脚本的进度输出和失败日志文件，便于重跑定位。当前还不是完整 APM 体系，但关键链路已经可追踪、可排错。
14. 【可答-需说明】怎么做的异步并行处理？
15. 知识图谱是怎么构建和使用的？
   - 项目现状版：为什么当前不做知识图谱？结构化表（事项/材料/流程）在本阶段的优势是什么？
16. 【可答-需说明】怎么做的 A/B 测试和 Prompt 版本管理？
17. 【可答-需说明】Token 计数和费用统计是怎么实现的？
18. 【可答】怎么做的数据预处理和数据清洗？
   - 真实回答：采集后先做字段标准化映射，把公开页面字段稳定写入 `gov_service/service_material/service_process`。数值字段（时限、到场次数）做类型清洗，材料和流程做结构化拆解，保留 `raw_payload` 便于追溯。导入用 source 唯一键去重，支持重复执行不产生脏重复。

## 第三类：性能优化类问题（12题）

1. 【可答-需说明】Token 成本是怎么控制的？
2. 【可答】延迟优化做了哪些措施？
   - 真实回答：主要做了三层优化：本地 embedding 模型常驻进程内复用；数据库侧用 ivfflat 并可调 probes；响应侧固定模板渲染，避免生成模型长推理。另外把本地模型设为 `local_files_only`，避免首请求被外网探测拖慢。
3. 【可答-需说明】缓存策略是什么？命中率多少？
   - 项目现状版：当前系统有哪些可落地缓存点（模型加载、查询结果等），如何逐步建立命中率指标？
4. 【可答-需说明】流式输出是怎么实现的？
5. 【可答-需说明】异步处理在哪些地方使用了？
6. 【可答】数据库查询是怎么优化的？
   - 真实回答：先做了 pgvector ivfflat 索引，再针对召回偏差引入 `probes` 可配置，保证“准确率/延迟”可调。我们还验证了执行计划，区分了 ANN 路径和精确排序路径的差异。结构化表上也建了常用索引和 source 唯一约束，保证导入和检索都稳定。
7. 【可答】模型推理速度是怎么优化的？
   - 真实回答：查询只做 embedding，不做长文本生成；模型由 `sentence-transformers` 懒加载后复用，热态请求明显更快。设备支持时走 CUDA，默认 `auto` 自动判定。同一模型用于回填和查询，避免在线做额外转换。
8. 【可答-需说明】前端渲染性能怎么优化？
9. 【可答】冷启动问题是怎么解决的？
   - 真实回答：我们识别到首请求卡顿来自模型加载和远端探测，于是改为本地文件优先（`GOVFLOW_EMBEDDING_LOCAL_FILES_ONLY=true`）并保持模型进程内缓存。这样冷启动成本主要发生一次，后续请求稳定。
10. 【可答-需说明】网络传输是怎么优化的？
11. 【可答】Embedding 计算是怎么优化的？
   - 真实回答：离线回填走批处理（batch），在线查询走单条快速编码；两者统一 768 维。回填脚本支持 `--device`、`--batch-size`、`--local-files-only`，可按机器资源调。并且全库已回填完成，线上不需要再做历史向量补算。
12. 【可答-需说明】怎么做的资源监控和自动扩缩容？
    - 项目现状版：在单机/小规模部署下，资源监控和容量预案该如何设计？何时需要自动扩缩容？

## 第四类：故障处理类问题（12题）

1. 【可答-需说明】Agent 死循环怎么检测和防止的？
2. 【可答-需说明】模型幻觉怎么控制的？
3. 【可答】模型超时怎么处理？
   - 真实回答：我们把超时分成“外部 API 超时”和“本地模型阻塞”两类处理。外部 API 有 timeout 配置；本地模型通过本地缓存优先和失败回退避免长时间挂起。出现异常时会返回可解释错误，而不是一直等待。
4. 【可答】检索结果不准确怎么办？
   - 真实回答：先用 `debug_vector_rank.py` 复现并看 TopN 距离，再看 SQL 执行计划判断是模型问题还是索引召回问题。我们就遇到过 ANN 误召回，最终通过 `ivfflat.probes` 调优修正。必要时再加关键词约束或阈值策略，避免错误强命中。
5. 【可答-需说明】工具调用失败怎么处理？
6. 【可答-需说明】多 Agent 之间通信失败怎么处理？
7. 【可答-需说明】知识库数据不一致怎么处理？
8. 【可答-需说明】高并发下的限流降级策略是什么？
9. 【可答-需说明】数据丢失和恢复策略是什么？
10. 【可答-需说明】模型输出格式异常怎么处理？
11. 【可答-需说明】敏感信息泄露怎么防止？
12. 【可答】系统容量规划是怎么做的？
   - 真实回答：当前按“小规模生产前”规划：先保证正确性和可观测，再按压测数据扩容。容量关注三块：API worker、embedding 推理资源（CPU/GPU/内存）、数据库连接池与索引性能。达到阈值后优先做读写分离或服务拆分，而不是盲目堆模型。
